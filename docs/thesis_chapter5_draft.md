## 5.2 测试方案设计

### 5.2.1 测试目标

本章测试从四个维度进行：

1. 功能完整性：验证“发现-识别-匹配-评估-报告-存储”链路是否打通。
2. 识别有效性：验证 `nmap -sV` 对开放端口服务及版本的识别能力。
3. 匹配有效性：验证规则库能否将服务指纹关联到对应 CVE。
4. 输出可用性：验证系统能否生成结构化 HTML 报告并支持历史记录查询。

### 5.2.2 测试用例设计

为覆盖“真实网络场景 + 可控复现场景”，设计如下测试用例：

- 用例 A（真实网络基线扫描）  
  目标网段：`172.26.204.0/24`；  
  扫描方式：`icmp,syn`（分层发现，syn 用于仅开放端口主机）；  
  端口：`22,80,443,445,3306,3389,5432,6379,8080`。

- 用例 B（本地靶场复现实验）  
  目标网段：`127.0.0.1/32`；  
  扫描方式：`icmp,syn`（分层发现，syn 用于仅开放端口主机）；  
  端口：`2222,6379`；  
  预期：识别 2 个服务并命中 2 条漏洞规则。

- 用例 C（风险模型改进对比实验）  
  在用例 B 的同一环境下，分别执行：  
  1）无资产画像扫描；  
  2）加载资产画像扫描（`--asset-profile docs/asset_profile.example.json`）；  
  对比风险分变化，验证新模型中 `AssetCriticality` 因子的影响。
  
说明：端口与服务识别由 `nmap -sV` 阶段完成，`--ports` 仅用于控制 nmap 的扫描范围；SYN 仅用于主机存活补充（需 root 或 `CAP_NET_RAW`）。

## 5.3 测试过程与结果

### 5.3.1 测试命令

（1）真实网络扫描命令：

```bash
python3 main.py scan \
  --target 172.26.204.0/24 \
  --methods icmp,syn \
  --ports 22,80,443,445,3306,3389,5432,6379,8080 \
  --name round1_formal
```

（2）本地靶场扫描（无资产画像）：

```bash
./lab/run_demo_scan.sh demo_lab_v2_no_profile
```

（3）本地靶场扫描（加载资产画像）：

```bash
./lab/start_demo_lab.sh

  --methods icmp,syn \
  --ports 2222,6379 \

  --name demo_lab_v2_with_profile
./lab/stop_demo_lab.sh
```

（4）历史记录查询命令：

```bash
python3 main.py history --limit 20
```

在异常依赖场景下（例如未安装 nmap，或目标服务响应特征不足导致版本字段缺失），系统将采用 socket 退化识别或从有限指纹信息中尽可能抽取产品信息。此时系统采用“置信度分级+手动确认”策略：若漏洞规则的 `version_rule` 为非通配表达（非 `*`），系统无法对版本条件进行验证，会将该条命中的匹配置信度强制降为 `LOW`，并在报告明细表中以“需要手动确认漏洞”进行标注。该机制用于显式区分“可自动判定的命中”与“信息不足的候选命中”，提示测试与演示过程中应补充版本信息或进行人工复核。

| 主机IP | 端口 | 服务 | CVE | 严重等级 | CVSS | 可利用性 | 匹配置信度 | 资产重要性 | 风险分 | 风险等级 |

### 5.3.5 风险模型改进对比结果（扫描ID=16 vs 17）

为验证 `AssetCriticality` 对量化结果的影响，选取同一靶场与同一漏洞集，对比“无资产画像（ID=16）”和“加载资产画像（ID=17）”。

表5-5 模型改进对比结果

| CVE | 端口 | 风险分(ID=16) | 风险分(ID=17) | 变化量(17-16) | 结论 |
|---|---:|---:|---:|---:|---|
| CVE-2021-41617 | 2222 | 7.68 | 7.48 | -0.20 | 画像将资产重要性从 8.0 调整到 7.0，风险分下降 |

该结果说明：在漏洞本身不变的情况下，风险模型能够根据资产画像动态调整风险分，满足“同漏洞在不同资产场景风险不同”的工程目标。

## 5.4 结果分析

### 5.4.1 功能达成度分析

结合测试结果，系统已实现任务书要求的核心能力：

1. 资产探测与发现：可完成目标网段存活主机发现与端口探测；
2. 服务识别与漏洞匹配：可识别服务版本并匹配本地规则库漏洞；
3. 风险评估与报告生成：可输出风险评分、风险等级和 HTML 可视化报告，并提供资产地图分组视图，便于按网段或资产组汇总查看风险分布；
4. 系统集成能力：支持 CLI 扫描、SQLite 存储和历史记录查询。

### 5.4.2 场景差异分析

真实网络场景（扫描 ID=6）中，发现主机但未识别到开放服务，故风险输出为 0。这一结果在工程上是合理的：当资产未暴露目标端口服务时，漏洞规则无法命中。  
可控靶场场景（扫描 ID=16/17）中，系统可稳定复现“2 服务、2 漏洞、2 中风险”结果，说明系统流程在已知输入条件下具备较好可重复性。  
同时，通过 ID=16 与 ID=17 的对比可见，新模型会受资产画像影响调整风险分，体现了风险评估对业务场景的自适应能力。

### 5.4.3 局限性分析

1. 规则库规模仍较小，当前主要用于验证流程与方法；
2. 版本识别结果受目标服务响应特征影响，存在识别精度边界；
3. 资产画像目前依赖人工维护，自动化资产分级机制仍需后续完善；
4. 风险模型尚未纳入网络拓扑纵深、攻击路径等更复杂因素。

## 5.5 本章小结

本章通过真实网络与本地可控靶场两类场景完成了系统测试。测试结果表明：系统能够完成从资产发现到报告输出的完整自动化流程，并在可复现实验中稳定识别服务与命中漏洞规则。综合来看，本系统已达到本科毕业设计原型的预期目标，为后续论文“总结与展望”中关于规则库扩展、模型优化与工程化部署提供了实证基础。
