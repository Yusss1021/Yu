# 10 分钟答辩讲稿（可直接使用）

本讲稿用于本科毕设现场陈述，按“页码/时间”组织，配合你的 PPT 使用。

本版本补齐了 10 分钟现场演示脚本，所有涉及数据库的命令都显式固定为 `--db data/scans.db`。
报告输出是目录包（report bundle），打开路径是 `reports/<name>/report.html`，同目录下必须保留 `assets/`。

## 0. 演示复现命令（复制即用）

说明：建议先 `cd intra_vuln_assessor`，确保 `data/scans.db` 写入位置稳定。

### 0.1 启动 Demo Lab（本机靶场，无需联网）

```bash
cd intra_vuln_assessor
./lab/start_demo_lab.sh

# 可选：确认当前目录与 DB 路径正确
python3 main.py history --db data/scans.db --limit 1
```

提示：`./lab/start_demo_lab.sh` 结束时会打印一个示例 scan 命令。现场演示请按下方 0.2 的命令执行，并且务必带 `--db data/scans.db`。

### 0.2 执行一次扫描（写入固定 DB）

```bash
cd intra_vuln_assessor
python3 main.py scan \
  --target 127.0.0.1/32 \
  --methods icmp,syn \
  --ports 2222,6379 \
  --name demo_lab_scan_round1 \
  --db data/scans.db
```

### 0.3 打开报告（report bundle 路径，不是单文件）

```bash
cd intra_vuln_assessor
xdg-open reports/demo_lab_scan_round1/report.html
```

备注：离线分享时请复制整个目录 `reports/demo_lab_scan_round1/`，不要只拷 `report.html`。

### 0.4 启动 Web（本地离线控制台）

```bash
cd intra_vuln_assessor
python3 main.py web \
  --host 127.0.0.1 \
  --port 5000 \
  --max-concurrent 3 \
  --db data/scans.db
```

浏览器打开：`http://127.0.0.1:5000`

## 1. 逐页讲稿（10 分钟）

## 1.1 第 1 页 背景与问题（0:00-0:50）

企业内网安全管理常见痛点是：资产不清、漏洞分散、修复优先级不明确。  
只做一次扫描，往往回答不了“先修什么、风险变化如何”。  
本课题目标是实现一个覆盖扫描、评估、对比的闭环系统，并能离线生成报告用于汇报。

## 1.2 第 2 页 研究目标（0:50-1:30）

本系统实现 6 个核心目标：

1. 资产发现；
2. 服务识别；
3. 漏洞匹配；
4. 风险量化；
5. 报告生成；
6. 历史对比与追踪。

## 1.3 第 3 页 系统架构（1:30-2:20）

系统分为三层：

1. 采集层：资产发现与服务识别；
2. 分析层：漏洞匹配与风险评估；
3. 展示层：报告、历史、Web 前端。

主流程由编排器统一调度，形成“发现-识别-匹配-评估-存储-对比”闭环。

## 1.4 第 4 页 资产发现与服务识别（2:20-3:20）

资产发现支持 `icmp/arp/syn` 组合策略。  
服务识别优先使用 `nmap -sV` 解析服务与版本。  
环境无 `nmap` 时会回退 socket 识别，保证系统可跑，但版本信息可能缺失，这会影响漏洞结论的确定性。

## 1.5 第 5 页 漏洞匹配设计（3:20-4:20）

漏洞匹配采用多条件约束，不是单关键词命中。  
匹配维度包括：`service`、`product`、`port`、`version_rule`。  
版本规则支持比较符和区间，用来降低误报。

## 1.6 第 6 页 风险量化模型（4:20-5:20）

当前模型公式：

`Risk = 0.45*CVSS + 0.20*AssetCriticality + 0.15*PortExposure + 0.10*ExploitMaturity + 0.10*MatchConfidence`

相比只看 CVSS，模型会把资产价值、暴露面和匹配置信度一起纳入，输出更接近“先修什么”的排序。

## 1.7 第 7 页 规则库管理（5:20-6:00）

系统支持两种规则更新方式：

1. 手动导入（适合论文主实验，复现更稳定）；
2. 自动更新（体现工程扩展能力）。

答辩演示里我以离线可复现为主，因此规则库以手动准备为主。

## 1.8 第 8 页 对比分析与并发（6:00-7:00）

系统支持两次扫描对比，输出：

1. 服务新增/消失/持续；
2. 漏洞新增/修复/持续；
3. 持续漏洞风险分变化。

并发能力包括引擎内部并发与 Web 任务级并发。数据库使用 SQLite，工程上通过 `WAL` 与 `busy_timeout` 处理并发写入压力。

## 1.9 现场演示（报告 + Web）（7:00-9:30）

下面我用本机 Demo Lab 跑一遍完整链路，整个过程不依赖外部网站。

### 7:00-7:30 运行扫描，生成报告与历史

我在终端执行一次扫描，数据库显式固定为 `--db data/scans.db`，便于复现和回看。

### 7:30-8:10 打开报告，指向「资产地图（分组视图）」

我打开报告目录包里的 HTML，路径是：`reports/demo_lab_scan_round1/report.html`。  
在报告中找到「资产地图（分组视图）」，这里按资产分组展示发现结果，先让评委快速看到资产面。

### 8:10-8:50 展示「需要手动确认漏洞」，解释原因（版本缺失，保守策略）

接着我在漏洞结果里指出提示语：`需要手动确认漏洞`。  
这类提示通常出现在两种情况叠加时：服务版本缺失，规则又需要做版本比较。系统不会武断下结论，而是用保守策略把不确定性标出来，提示人工去核对真实版本。

### 8:50-9:10 说明离线特性（assets/ + 无外部请求）

报告不是单个 HTML 文件，而是目录包：同目录下有 `assets/`。  
我现场展示 `assets/` 里包含图表脚本，例如 `assets/chart.umd.min.js`，所以报告可离线打开，也不会向外部域名发起请求。分享时复制整个 `reports/demo_lab_scan_round1/` 目录即可。

### 9:10-9:30 打开 Web 扫描详情，确认同样包含「需要手动确认漏洞」

最后我打开本地 Web 前端进入 scan detail 页面。页面里有注意事项说明，同时在漏洞表中同样会出现 `需要手动确认漏洞`，做到 CLI 结果与 Web 展示一致。

## 1.10 第 10 页 总结与展望（9:30-10:00）

本课题完成了“扫描-评估-报告-历史-对比”的闭环，并且支持离线报告展示与本地 Web 演示。  
后续我计划补齐规则源标准化与误报评估数据集，同时继续增强趋势分析与告警联动能力。

## 2. 高频追问与标准回答

## 2.1 创新点是什么？

把“发现-识别-评估-对比”做成可运行闭环，并将风险量化与历史变化结合，最终能落到可演示的报告与对比结论。

## 2.2 为什么不只用 CVSS？

CVSS 只描述漏洞本身，不描述业务语境。  
系统引入资产重要性、暴露面和匹配置信度，使结果更符合真实修复优先级。

## 2.3 报告为什么是目录包，不能只拷贝 report.html？

报告路径是 `reports/<name>/report.html`，同目录必须带 `assets/`。  
例如 `assets/chart.umd.min.js` 是图表脚本，单拷 HTML 会导致静态资源缺失，图表不完整。

## 2.4 「需要手动确认漏洞」是什么意思？

它表示系统当前证据不足以做确定性判断，常见原因是服务版本缺失，规则又需要版本比较。  
提示的目的不是吓人，而是把不确定性显式给出来，提醒通过人工核对版本或二次验证。

## 2.5 如何降低误报？

通过多条件匹配和版本规则约束，避免单字段命中。  
同时输出 `match_confidence` 辅助人工判定，并对不确定项给出“需要手动确认漏洞”的提示。

## 2.6 规则如何更新？

支持手动导入与自动更新。  
主实验推荐手动精选规则以保证复现，自动更新用于展示扩展能力。

## 2.7 并发会不会导致数据库冲突？

存在高并发写入压力，因此采用了 SQLite `WAL` 与 `busy_timeout` 优化。  
实践建议 `--max-concurrent` 设置在 `2~4`。

## 2.8 系统局限性是什么？

规则覆盖仍有限，深度 POC 验证能力尚未全面集成。  
项目定位是本科毕设可落地原型，强调可解释与可演示。

## 2.9 一句话总结贡献

在本科范围内实现了可运行、可解释、可演示、可离线分享的内网脆弱性评估原型系统。
