# 第4章 系统详细设计与实现

本章在第3章总体设计基础上，给出系统各功能模块的详细实现方案。系统采用 Python 语言开发，围绕“资产发现、服务识别、漏洞匹配、风险评估、报告生成、结果存储与历史对比”构建完整自动化流程。

## 4.1 系统实现架构

### 4.1.1 分层结构

系统实现采用“入口层 + 编排层 + 功能层 + 数据层”四层结构：

1. 入口层（CLI）  
   负责参数接收与命令分发，支持 `scan`、`history`、`compare` 三类命令。

2. 编排层（Orchestrator）  
   负责统一调度业务流程，串联资产发现、服务识别、漏洞匹配、风险评估、报告生成与结果入库。

3. 功能层（Scanners / Vuln / Risk / Report）  
   按模块实现具体业务算法。

4. 数据层（SQLite Repository）  
   提供扫描记录、资产、服务、漏洞数据的持久化与历史差异分析能力。

### 4.1.2 端到端执行流程

系统一次扫描任务的主流程如下：

```text
输入目标网段与参数
-> 资产发现(icmp/arp/syn，逐层补充)
-> 服务识别(nmap -sV 或 socket fallback)
-> 漏洞规则匹配(服务+产品+版本)
-> 风险评分与分级
-> HTML 报告生成
-> SQLite 持久化
-> 返回摘要统计
```

## 4.2 核心数据模型设计

系统通过 4 个核心数据模型在模块间传递结构化数据：

1. `HostAsset`：资产实体，字段包括 `ip`、`mac`、`discovered_by`、`open_ports`。
2. `ServiceFingerprint`：服务指纹，字段包括主机、端口、协议、服务名、产品、版本、额外指纹信息。
3. `VulnerabilityFinding`：漏洞命中结果，字段包括 `cve_id`、`severity`、`cvss`、修复建议等。
4. `RiskFinding`：在漏洞结果基础上追加 `risk_score` 与 `risk_level`。

该设计使各模块之间低耦合：扫描模块只关注资产与服务，匹配模块只关注规则命中，评估模块只关注风险量化。

## 4.3 资产发现模块设计与实现

### 4.3.1 模块目标

资产发现模块负责在目标网段中识别可达主机并补充 MAC 等信息，为后续服务识别提供输入。端口与服务识别由 `nmap -sV` 阶段完成。该模块由 `AssetDiscoveryEngine` 实现，支持三种发现方式：

1. ICMP 探测：调用系统 `ping` 命令进行主机存活判断；
2. ARP 探测：在具备 `scapy` 和权限条件下进行二层广播发现；
3. SYN 半开探测：对未被 ICMP/ARP 发现的主机进行 SYN 探测，判断是否存在开放端口（需 root 或 `CAP_NET_RAW`，依赖 `scapy`）。

### 4.3.2 关键实现要点

1. 网段展开  
   使用 `ipaddress.ip_network(cidr, strict=False)` 展开主机地址集合。

2. 并发扫描  
   对 ICMP/ARP/SYN 探测均采用 `ThreadPoolExecutor`，提高扫描效率。

3. 结果归并  
   资产条目通过 IP 聚合，同一主机可同时保留多种 `discovered_by` 标签。

4. 兼容性降级  
   当 ARP 所需环境不满足时（无 scapy 或无权限），ARP 返回空结果，不影响整体流程继续执行。

## 4.4 服务识别模块设计与实现

### 4.4.1 识别策略

服务识别模块由 `ServiceFingerprintEngine` 实现，采用“优先高精度、失败可退化”的双路径策略：

1. 优先路径：调用 `nmap -sV` 获取服务与版本指纹；
2. 退化路径：若未安装 nmap，则通过 socket 连通性与常见端口映射给出基础服务识别。

### 4.4.2 Nmap 识别实现

1. 每个目标主机构造命令：

```text
nmap -sV -Pn --version-intensity 5 -p <ports> -oX - <ip>
```

2. 输出采用 XML 格式，使用 `xml.etree.ElementTree` 解析；
3. 仅保留 `state=open` 端口；
4. 当 nmap 未给出 `version` 字段时，系统会从 `product/extrainfo` 中用正则提取版本号，提升匹配命中率；
5. 针对服务探测耗时问题，设置了按端口规模自适应的扫描超时窗口，避免识别结果提前丢失。

### 4.4.3 去重与标准化

识别结果以 `(host_ip, port, protocol)` 作为唯一键去重，确保后续匹配与入库阶段不会产生重复条目。

## 4.5 漏洞规则库与匹配引擎实现

### 4.5.1 规则库结构

本地规则库采用 JSON 格式，每条规则至少包含以下字段：

1. `cve_id`
2. `service`
3. `product`
4. `version_rule`
5. `severity`
6. `cvss`
7. `description`
8. `remediation`

规则库文件位于 `vuln_assessor/vuln/rules.json`，便于后续扩展为批量导入 CVE/CNVD 数据。

### 4.5.2 匹配逻辑

匹配引擎 `VulnerabilityMatcher` 对每个服务指纹执行两阶段判断：

1. 服务/产品匹配  
   - 若规则定义了 `service`，要求其为识别服务名子串；  
   - 若规则定义了 `product`，要求其为识别产品名子串；  
   - 可选 `port` 字段可进一步约束端口。

2. 版本匹配  
   支持以下版本规则表达：
   - 范围表达：`2.4.49-2.4.50`
   - 比较表达：`<=8.8`、`>=1.2.0, <2.0.0`、`==6.2.5`
   - 通配表达：`*`

版本字符串统一抽取数字后转为元组比较，例如 `7.4` -> `(7, 4)`。

### 4.5.3 匹配置信度分级与手动确认策略

为提升漏洞命中结果的可解释性，并降低“版本信息不足”导致的误报风险，系统在规则命中阶段为每条漏洞结果计算匹配置信度（`match_confidence`），并将其映射为分级标签用于报告展示。该策略用于减少误报并保持结论保守，在证据不足时降低自动判定强度，并提示人工复核。

1. 置信度分级规则

系统将 `match_confidence` 映射为三档：

- HIGH：`match_confidence >= 7.5`
- MEDIUM：`5.0 <= match_confidence < 7.5`
- LOW：`match_confidence < 5.0`

2. “需要手动确认漏洞”策略

当服务指纹未能提取到有效版本信息（`version` 缺失），且命中的规则条目 `version_rule` 为非通配表达（非 `*`）时，系统将该条命中结果的置信度强制降为 `LOW`，并在 HTML 报告与 Web 展示中提示“需要手动确认漏洞”。该提示表示：当前命中仅完成服务或产品层面的关联，版本条件无法被验证，需在补充版本信息后由安全人员进行复核。

## 4.6 风险评估模型实现

### 4.6.1 模型设计

本课题采用轻量化多因子评分模型，将漏洞严重度、资产重要性、暴露面与匹配可靠性结合，计算综合风险分：

```text
RiskScore = 0.45*CVSS + 0.20*AssetCriticality + 0.15*PortExposure + 0.10*ExploitMaturity + 0.10*MatchConfidence
```

其中：

1. `CVSS`：漏洞基础评分；
2. `AssetCriticality`：资产重要性，可由资产画像文件按 IP 注入；
3. `PortExposure`：按端口暴露风险预设权重；
4. `ExploitMaturity`：漏洞可利用性，来自规则字段或按严重等级缺省映射；
5. `MatchConfidence`：匹配置信度，按服务/产品/版本识别完整度估计。

最终得分截断到 `[0, 10]` 区间，并按阈值分级：

1. `score >= 8.0` -> `HIGH`
2. `5.0 <= score < 8.0` -> `MEDIUM`
3. `score < 5.0` -> `LOW`

### 4.6.2 模型实现特征

1. 规则与模型解耦：漏洞匹配产出与风险评分可独立演进；
2. 参数可解释：每一项风险分可追溯到 CVSS、资产重要性、暴露面、可利用性与匹配置信度；
3. 工程可扩展：后续可引入网络拓扑位置、资产业务价值等额外特征。

## 4.7 报告生成模块实现

### 4.7.1 生成流程

报告生成由 `HtmlReportGenerator` 完成，核心步骤如下：

1. 汇总统计指标（主机数、服务数、风险总数、高中低分布）；
2. 截取 Top 风险列表；
3. 组织图表数据（风险等级分布）；
4. 通过 Jinja2 模板渲染报告主页面并按任务名称输出为报告包：主页面位于 `reports/<name>/report.html`，静态资源输出到同级 `reports/<name>/assets/` 目录，确保在离线环境中可直接打开与浏览。

### 4.7.2 报告内容结构

报告模板 `report.html.j2` 输出以下信息：

1. 扫描元信息：目标网段、扫描方法、扫描端口、生成时间；
2. 总览卡片：主机/服务/风险数量及高中低统计；
3. 风险明细表：主机、端口、服务、CVE、CVSS、风险分、修复建议；
4. 可视化图表：使用 Chart.js 绘制风险等级分布、端口 Top10、置信度分布等统计图；
5. 资产明细与资产地图：展示 IP、MAC、发现方式、开放端口，并提供按网段聚合的分组视图。

### 4.7.3 资产地图（分组视图）输出设计

为便于在较大网段扫描结果中快速定位资产分布与暴露面，报告模块在资产明细表之外增加“资产地图（分组视图）”。该视图定位为统计视图，以网段为单位聚合资产，并给出发现方式、端口与置信度等统计图表，便于运维复盘与演示展示。

1. 网段分组规则

对 IPv4 资产，系统默认按 `/24` 网段聚合；当扫描目标前缀长度大于 `/24`（例如 `/32`、`/28`）时，分组粒度切换为目标 CIDR，以避免生成与目标范围无关的更大分组。该规则可等价表达为分组前缀长度 `group_prefix = max(24, target_prefix)`。对非 IPv4 或无法解析的目标，统一归入 `UNKNOWN` 分组以保持统计口径完整。

2. 主发现方式（primary discovered_by）选择

同一资产可能同时被 `icmp`、`arp`、`syn` 多种方式发现。为保证分组统计口径一致，系统为每个资产选取一个主发现方式，优先级为 `icmp > arp > syn`。

3. 图表输出

- 发现方式分布：按资产主发现方式（`icmp`/`arp`/`syn`）统计数量，用于解释资产发现路径构成。
- 端口 Top10：统计本次扫描中开放端口的出现次数并展示 Top10，用于刻画暴露面集中度。
- 置信度分布：按 HIGH、MEDIUM、LOW 三档统计漏洞命中结果数量，直观反映自动判定与需要人工复核的比例。

## 4.8 数据存储与历史对比实现

### 4.8.1 数据库表设计

系统使用 SQLite 作为本地存储，主要包含 4 张表：

1. `scans`：扫描任务元数据与统计信息；
2. `assets`：每次扫描发现的资产明细；
3. `services`：服务识别结果；
4. `vulnerabilities`：漏洞与风险评估结果。

该结构满足“一次扫描多资产、多服务、多漏洞”的一对多关系建模需求。

### 4.8.2 持久化过程

每次扫描完成后，按以下顺序写入数据库：

1. 写入 `scans` 主记录，获取 `scan_id`；
2. 关联写入 `assets`；
3. 关联写入 `services`；
4. 关联写入 `vulnerabilities`；
5. 事务提交。

### 4.8.3 历史差异对比

对比功能以 `(host_ip, port, cve_id)` 作为漏洞唯一标识，计算两次扫描集合关系：

1. 新增漏洞：`new_set - base_set`
2. 已修复漏洞：`base_set - new_set`
3. 持续存在漏洞：`base_set ∩ new_set`

该机制可用于周期扫描中的风险治理跟踪。

## 4.9 命令行交互与系统编排实现

### 4.9.1 CLI 设计

命令行入口支持三个子命令：

1. `scan`：执行扫描任务；
2. `history`：查看历史扫描记录；
3. `compare`：对比两次扫描差异。

其中 `scan` 支持参数化配置目标网段、扫描方法、端口范围、报告路径、数据库路径和任务名称，满足实验与演示场景下的灵活调用需求。

### 4.9.2 编排器实现

`ScanOrchestrator` 负责系统全流程调度，核心执行顺序为：

1. `discover()` -> 资产发现；
2. `fingerprint()` -> 服务识别；
3. `match()` -> 漏洞匹配；
4. `evaluate()` -> 风险评估；
5. `generate()` -> 报告生成；
6. `save_scan()` -> 结果持久化；
7. 返回统计摘要（扫描ID、时长、风险分布等）。

编排器使系统具备单一流程入口，便于后续加入任务调度、Web 化接口或并行任务管理能力。

## 4.10 本章小结

本章详细介绍了系统从架构分层到模块实现的完整技术路径。通过 `CLI + 编排器 + 功能模块 + SQLite` 的工程化设计，系统实现了从资产发现到风险报告输出的自动化闭环。该实现既满足任务书提出的核心功能要求，也为第5章实验验证与结果分析提供了可重复、可追溯的实现基础。
