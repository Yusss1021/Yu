# 安全与伦理说明（必读）

> 本项目定位：企业内部网络的**脆弱性扫描与风险评估**工具。
> 
> 关键词：**评估 / 提示 / 风险优先级**（不是攻击工具）。

## 1) 使用前必须满足的条件（授权 / 范围 / 时间窗口 / 通知）

在开始任何扫描前，请同时满足以下条件：

1. **明确授权**：
   - 你必须获得目标资产所有者/管理方的**书面或可追溯授权**。
   - 未授权的公网、第三方主机、同网段“顺手扫一下”都属于禁止范围。

2. **明确范围**：
   - 只在授权的网段/主机/端口范围内运行。
   - 建议把范围写成可核对的清单（例如：`192.168.1.0/24` + 白名单端口）。

3. **明确时间窗口**：
   - 选择业务低峰期或维护窗口，避免影响生产。
   - 如需连续多轮对比扫描，提前约定频率（例如每天一次/每周一次）。

4. **明确通知与沟通**：
   - 告知网络/安全/运维相关人员：扫描时间、来源 IP、工具行为（ICMP/ARP/SYN、nmap 指纹）。
   - 约定告警处理方式（例如 IDS 告警出现时谁确认、是否需要临时白名单）。

> 最低要求：**授权 + 范围 + 时间窗口**缺一不可；建议额外完成通知。

## 2) 扫描可能的影响（你需要提前知道）

即便是“评估型扫描”，也可能产生副作用：

1. **触发 IDS/防火墙/日志告警**：
   - ICMP、端口探测、服务指纹识别都可能留下明显日志。
   - 这不是“被发现了”，而是正常现象；重点是提前沟通，避免误判为真实入侵。

2. **端口探测带来的服务压力**：
   - 在端口较多、频率较高、目标较弱（老旧设备/嵌入式/工控）时，可能出现短暂抖动。
   - 解决思路：小范围试跑 → 观察 → 再扩大范围。

3. **ARP 探测的网络副作用**（局域网二层场景）：
   - ARP 广播会增加局域网广播流量。
   - 某些设备可能因 ARP 行为更新或刷新缓存，导致日志增多或短暂异常感知。

4. **SYN/半开探测的敏感性**：
   - SYN 探测更“激进”，更容易触发防护设备策略或告警。
   - 非必要不要上来就开 SYN；优先用 `icmp`，在授权与维护窗口内再逐步增强（`icmp → arp → syn`）。

## 3) 数据与隐私（SQLite / 报告敏感信息 / 保管与脱敏）

### 3.1 数据会保存在哪里？

- 扫描结果默认会写入 SQLite 数据库：`data/scans.db`。
- 报告会生成离线可打开的目录包：`reports/<name>/report.html` + `assets/`。

> 重要提示：数据库路径常见 gotcha 是“相对路径”。建议固定使用 `--db data/scans.db`，并在 `intra_vuln_assessor/` 目录下运行。

### 3.2 数据可能包含哪些敏感信息？

扫描与报告可能包含（或推断出）：

- IP / 网段 / 主机在线情况
- 开放端口、服务名称与**服务版本**（指纹信息）
- 漏洞匹配结果（CVE/规则命中、风险等级、置信度提示）

这些信息可能构成“资产画像”，应按敏感数据对待。

### 3.3 如何保管与脱敏（建议做法）

1. **最小化存储**：只保留必要的扫描轮次与报告，避免长期堆积。
2. **访问控制**：
   - `data/scans.db` 与 `reports/` 目录建议限制权限（仅项目成员可读）。
   - 共享报告时，确认接收方有授权与知情。
3. **脱敏共享**（对外演示/论文截图时尤其重要）：
   - 隐去真实 IP/网段、主机名、资产标识；可用 `10.0.0.x` 之类占位。
   - 不展示可被直接用于定位的敏感端口组合与详细指纹；保留“方法、流程、现象、统计结果”。
4. **留存审计**：建议记录扫描的授权材料、范围、时间与操作者，便于追溯。

## 5) 建议的安全操作（降低风险、提高可控性）

1. **先小范围试跑**：先选 1~3 台测试主机或一个小网段验证流程与影响，再扩大范围。
2. **低频率、分批次**：避免高并发/高频扫描；多轮对比时固定频率并在低峰执行。
3. **端口白名单优先**：
   - 优先扫描业务相关端口（白名单），避免“全端口”带来的压力与误告警。
   - 端口范围必须可解释、可回溯。
4. **避免生产峰值**：不要在业务高峰或变更窗口叠加扫描。
5. **遵循递进策略**：`icmp → arp → syn`，每增加一种方法都要重新评估影响。
6. **正确理解结论**：
   - “扫描不到 ≠ 不存在风险”（ICMP/ARP 不通、权限不足会漏报）。
   - “匹配到 CVE ≠ 一定存在漏洞”（版本缺失/指纹偏差会有误报风险，需人工确认）。
