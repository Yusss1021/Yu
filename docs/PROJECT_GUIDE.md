# 项目指南（零基础 / 论文友好版）

> 读者画像：你可能**完全零基础**，第一次接触“内网扫描/端口扫描/漏洞匹配”，也不熟悉 Nmap、ICMP、ARP 等概念。
> 
> 本文目标：用**最少的前置知识**带你理解本项目“从输入一个网段，到输出一份可读报告并可历史对比”的**端到端链路**。

---

## 1. 项目概览

### 1.1 一句话目标

在授权的企业内网/实验网中，自动完成：**资产发现 → 服务识别 → 漏洞匹配 → 风险评估 → 报告生成 → 入库与历史对比 / Web 展示**。

### 1.2 四大模块（对应开题报告）

本项目可以按“开题报告”口径拆成四大模块（也是论文结构最常用的写法）：

1. **资产探测与发现模块**：复合扫描引擎（ICMP/ARP/TCP SYN 递进）。
2. **服务识别与漏洞匹配模块**：优先调用 `nmap -sV` 获取服务与版本；无 nmap 时 socket 回退；再用本地规则库做 CVE 匹配。
3. **风险评估与报告生成模块**：基于简化风险模型输出 HIGH/MEDIUM/LOW，并生成 HTML 报告。
4. **系统集成与应用模块**：CLI/Web 入口、SQLite 持久化、历史查询与对比。

### 1.3 端到端链路图（论文可直接引用）

下面是一次扫描从命令行入口到产出结果的固定链路（**按执行顺序**）：

```
discover  →  fingerprint  →  match  →  risk  →  report  →  storage  →  compare / web
资产发现     服务识别        漏洞匹配     风险评估   报告生成   入库保存      历史对比/前端
```

你可以把它理解为一条“流水线”：上一步的输出是下一步的输入。

## 3. 复合扫描引擎（ICMP/ARP/SYN）

### 3.1 为什么要“递进式复合扫描”？

单一发现手段在真实网络中经常失效：

- 有的主机禁 ping（ICMP 被屏蔽），但其实在线。
- 有的环境在二层能看到 ARP 响应，但三层 ICMP 不通。
- 有的主机对 ICMP/ARP 都不响应，但**某些端口**对外开放（例如只开了 TCP 服务）。

因此本项目采用“逐层补充”的递进策略：

1. 先用 **ICMP** 快速扫一遍（成本低、噪声相对小）。
2. 再用 **ARP** 在局域网补一遍（更贴近二层真实在线情况）。
3. 最后用 **SYN 半开**补齐“仅开放端口的主机”（更激进、对权限要求更高）。

并且：**SYN 只对前两种没发现的主机执行**，避免无意义的噪声与耗时。

### 3.2 三种发现方式的权限要求与局限

#### ICMP（网络层）

- 做什么：对目标 IP 发 ping（ICMP Echo）。
- 优点：简单、快速；不需要额外依赖。
- 局限：很多内网/防火墙会禁 ping，导致“在线但探测不到”。
- 权限：通常普通用户即可（取决于系统策略与 ping 实现）。

#### ARP（数据链路层，仅局域网）

- 做什么：发送 ARP 请求，获得局域网内主机的响应与 MAC。
- 优点：在同一二层网络内常常比 ICMP 更可靠。
- 局限：只能用于局域网二层；跨网段/路由场景 ARP 不适用。
- 权限与依赖：
  - 通常需要 **root 权限**。
  - 依赖 `scapy`。

#### SYN（TCP 半开端口发现）

- 做什么：对指定端口发 TCP SYN，观察 SYN-ACK 来判断“该主机在该端口有服务”。
- 优点：能发现“禁 ICMP/ARP，但端口开放”的主机。
- 局限：
  - 需要目标端口列表（由 `--ports` 指定）。
  - 易受防火墙/IDS 策略影响；噪声更大。
- 权限与依赖：
  - 需要 **root 或 `CAP_NET_RAW`**。
  - 依赖 `scapy`。
  - 如果缺依赖/权限不足，本项目会继续流程，并尽可能从后续指纹阶段回填端口信息。

## 4. 服务识别与漏洞匹配

### 4.1 服务识别（fingerprint）

本项目优先使用 **Nmap** 进行服务识别：

- 核心命令形态：`nmap -sV -Pn -p <ports> -oX - <ip>`
- 会输出：服务名（如 ssh/http）、产品（product）、版本（version）等。

当系统未安装 `nmap` 时，会回退到：

- socket 探测端口开放性 + 常见端口服务映射（例如 22→ssh，6379→redis）
- 这种回退通常拿不到“准确版本”，因此后续漏洞匹配的置信度会更低（需要你重点理解第 5 节）。

### 4.2 漏洞匹配（match）

漏洞匹配基于本地规则库（JSON），按以下维度组合匹配：

- service / product / port / version_rule

其中版本规则 `version_rule` 支持：

- 通配：`*`
- 比较符：`< <= > >= ==`
- 区间：`x.y - z.w`

匹配输出包含：

- `match_confidence`（0~10）：匹配置信度
- `confidence_tier`：HIGH/MEDIUM/LOW
- `manual_confirmation_needed`：是否需要人工确认

## 5. 风险模型与置信度

### 5.1 风险模型（简化但可解释）

系统使用可解释的线性加权模型（报告中会展示公式），核心思想：

- CVSS 严重性越高 → 风险越高
- 资产越关键（资产画像/服务类型） → 风险越高
- 端口越暴露（常见高风险端口） → 风险越高
- 漏洞越“可利用”（exploit maturity） → 风险越高
- 漏洞匹配越可信（match confidence） → 风险越高

最终输出：

- `risk_score`（0~10）
- `risk_level`：HIGH / MEDIUM / LOW

### 5.2 置信度分级（必须按此口径写入论文）

系统将 `match_confidence` 映射为三档 `confidence_tier`：

- **HIGH**：`match_confidence >= 7.5`
- **MEDIUM**：`match_confidence >= 5.0`
- **LOW**：其他情况

### 5.3 “版本缺失 + 非通配版本规则” → 强制 LOW + 手动确认

这是论文/答辩中必须强调的一个“误报控制”机制：

如果 **服务版本缺失**（例如未安装 nmap、或目标服务未暴露版本）并且规则的 `version_rule` **不是**通配 `*`，那么系统会：

1. **强制 `confidence_tier = LOW`**
2. 标记 `manual_confirmation_needed = True`
3. 在风险建议中提示：**“需要手动确认漏洞”**

原因：

- 没有版本信息时，系统只能依据 service/product/port 进行粗匹配。
- 很多漏洞只影响特定版本区间；缺版本会显著提高**误报风险**。
- 因此必须用“低置信度 + 人工确认”把不确定性显式暴露出来。

## 6. 报告与资产地图（分组视图）

> 说明：本项目的“资产地图”是**分组视图**（不是网络拓扑图）。它用于论文中更清晰地表达“扫描到的资产分布”。

### 6.1 资产地图分组规则（必须写死为文档口径）

资产地图分组 key 的计算规则如下（严格按此写论文/说明）：

1. **IPv4：默认按 /24 分组**
   - 示例：`192.168.1.23` 归入 `192.168.1.0/24`
2. **若 target CIDR 比 /24 更小**（`prefixlen > 24`，例如 `/28`、`/32`）：
   - **使用 target 网段本身**作为唯一分组 key
   - 示例：target=`10.0.0.8/29`，则所有资产都归入 `10.0.0.8/29`
3. **非 IPv4：分组 key = `UNKNOWN`**
   - 并说明：当前仅支持 IPv4 的分组展示（IPv6 发现阶段会跳过 ARP/SYN）。

> 设计意图（论文友好）：/24 是最常见的“内网口径”，能把大网段的结果分桶，展示更直观；当 target 本身更细（如 /32 单主机演示）时，按 target 分组更符合读者直觉。

### 6.2 发现方式“主归属”（必须写死）

一台主机可能被多种方式发现（例如先 ICMP 再 SYN）。为了做“发现方式统计图”时不重复计数，本项目对每台主机定义一个**主归属**：

- 主归属优先级固定为：`icmp > arp > syn`

含义：

- 如果该主机的 `discovered_by` 同时包含 icmp/arp/syn，则统计时**只归到优先级最高**的那一类。

这样做的好处：

- 统计口径稳定（每台主机只算一次）
- 图表更容易解释（不会出现“总数对不上”的疑惑）

## 7. 数据存储与历史对比

### 7.1 默认数据库路径（重要 gotcha）

系统默认数据库路径是：

- `data/scans.db`

注意：这是一个**相对路径**。当你在不同目录执行命令时，可能会导致“写入了意外位置的 data/scans.db”。

论文实验与长期使用建议：固定 `--db`：

```bash
python3 main.py scan \
  --target 192.168.1.0/24 \
  --methods icmp,arp,syn \
  --ports 22,80,443,445,3306,3389 \
  --name round1 \
  --db /abs/path/to/scans.db
```

### 7.2 对比能力的意义

单次扫描只能回答“现在是什么样”。历史对比可以回答：

- 新增了哪些服务/漏洞？
- 哪些服务/漏洞消失（可能已加固）？
- 持续漏洞的风险分是否变化？

这也是论文里“系统价值”的常用论点。

## 8. 常见问题

### 8.1 扫描不到主机 = 目标不在线吗？

不一定。常见原因：

- ICMP 被屏蔽；
- 你不在同一二层网络导致 ARP 不适用；
- SYN 需要 root/`CAP_NET_RAW` 或缺少 `scapy`。

建议：

1) 先确认 target 合法且在授权范围内；
2) 优先用 demo lab 验证工具链路没问题；
3) 需要更强发现能力时再开 ARP/SYN 并确保权限与依赖。

### 8.2 为什么报告里提示“需要手动确认漏洞”？

因为该条漏洞匹配属于 **LOW 置信度**（常见原因是“版本缺失但规则要求版本区间”）。

这不是系统失败，而是系统在提醒你：

- 该条结论存在误报风险，应该通过人工手段（登录设备、查看真实版本、二次验证）确认。

### 8.3 为什么服务识别信息很少？

如果未安装 `nmap`，系统会走 socket 回退，只能拿到“端口开放”与少量 banner 信息，通常无法得到准确版本。

### 8.4 为什么 ARP/SYN 没生效？

ARP/SYN 往往需要：

- 安装 `scapy`
- 使用 root 或具备 `CAP_NET_RAW`

在权限不足或依赖缺失时，系统会给出警告，并继续后续流程。

### 8.5 常见误解澄清（建议写入论文 FAQ）

1. **“资产地图=网络拓扑”**：本项目资产地图是分组统计视图，不做拓扑推断。
2. **“扫描不到=不存在风险”**：扫描受环境影响；禁 ICMP/封端口/权限不足都会导致漏报。
3. **“匹配到 CVE 就一定有漏洞”**：规则匹配是基于指纹推断；版本缺失时必须人工确认。
4. **“HIGH 风险=一定马上被攻破”**：风险分是优先级排序工具，不等同于现实攻击必然性。

## 术语小词典

- **ICMP**：Internet Control Message Protocol，常见用途是 ping（探测主机是否可达）。
- **ARP**：Address Resolution Protocol，局域网内用来把 IP 解析为 MAC 的协议，可用于二层主机发现。
- **SYN**：TCP 三次握手的第一个包（SYN）。半开 SYN 探测可判断端口是否开放。
- **CVE**：Common Vulnerabilities and Exposures，公开漏洞编号体系（如 `CVE-2021-41617`）。
- **CVSS**：Common Vulnerability Scoring System，漏洞严重性评分标准（通常 0~10）。
- **Nmap**：常用网络扫描工具。本项目优先用它做 `-sV` 服务指纹识别。
